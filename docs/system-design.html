02
Flujo: Generar un Sistema
Desde que el usuario describe el sistema hasta que descarga el código generado.
1
Usuario crea proyecto
Describe el sistema en lenguaje natural, selecciona stack y agentes. Se crea el proyecto en estado idle.
POST /api/projects → status: idle
2
Spec Engine genera documentación
3 llamadas secuenciales a Claude: spec.md → data-model.md → api-design.md. El usuario ve el texto en tiempo real via SSE.
POST /api/projects/:id/spec/generate → GET /spec/stream (SSE)
3
Usuario revisa, edita y ejecuta
Revisa el spec generado, puede editarlo con @uiw/react-md-editor. Al hacer clic en "Iniciar", el job se encola en BullMQ.
POST /api/projects/:id/start → BullMQ job enqueued
4
Orchestrator ejecuta 6 capas secuencialmente
El worker BullMQ lee el spec y ejecuta: Database → Backend → Frontend → Testing → Docs → Deployment. Cada capa recibe contexto de las anteriores. Sin paralelismo en MVP.
BullMQ Worker → Orchestrator → Layer 1…6 sequential
5
Agentes trabajan con Tool Use
Cada agente usa Claude Tool Use en loop: envía prompt → Claude responde con createFile/readFile/listFiles → backend ejecuta tool → repite hasta taskComplete. Emite eventos WebSocket en tiempo real.
Claude API tool_use → createFile → WS emit → repeat
6
Proyecto completado
Al completar Layer 6, status cambia a done. Emite project:done via WebSocket. El usuario descarga el ZIP con toda la estructura.
project:done → GET /api/projects/:id/download → ZIP stream

Decisiones Técnicas
Decisiones clave tomadas durante el diseño. Cada una justificada contra la alternativa descartada.
Tema
Decisión
Por qué
Auth
JWT custom en Fastify
Eliminamos NextAuth.js v5 — Fastify es el único auth server. Cookies httpOnly.
Dashboard visual
Vanilla Canvas API
Total control sobre animaciones y partículas. Sin overhead de D3 o Three.js. Eventos manejados manualmente (hit-testing).
Ejecución de agentes
Secuencial 6 capas
Sin paralelismo en MVP para evitar inconsistencias. Post-MVP se evaluará.
IA / Agentes
Claude Tool Use (loop)
Agentes no generan texto plano — crean archivos iterativamente via tools.
API key storage
AES-256-GCM en user_settings
Cada usuario su propia key. No hay key del sistema. Encrypt con IV único.
Archivos generados
Filesystem + metadata en BD
Contenido en disco (no BLOB). BD solo path, size_bytes, agent. ZIP con archiver streaming.
Cola de trabajos
BullMQ worker separado
Worker como proceso independiente en Railway. Concurrency 3 proyectos simultáneos.
Rate limiting
Redis counters con TTL
Login: 5 intentos/15min. Registro: 3/hora/IP. API key verify: 5/hora.
Spec editor
@uiw/react-md-editor
Editor markdown ligero con preview. No Monaco (demasiado pesado para este caso).
UI Components
shadcn/ui + Tailwind CSS
Componentes copiados al proyecto (no dependencia). Customizables con Tailwind.
06
Eventos WebSocket
El servidor emite estos eventos al cliente en tiempo real via /ws/projects/:id. Auth via JWT cookie en handshake. Reconnect cada 3s con replay desde lastEventId.
agent:status
agentId, agentType (dba|backend|frontend|qa|docs|deploy), status (idle|working|done|error|paused), progress 0-100, currentTask
agent:log
agentId, agentType, type (info|ok|warn|error), message, timestamp ISO 8601
file:created
agentId, agentType, fileName, filePath, fileSize — aparece en panel de archivos con animación
project:progress
progress 0-100, layer 1-6, layerName — actualiza barra general y etiqueta de capa activa
project:done
totalFiles, tokensInput, tokensOutput, duration (segundos) — activa botón descarga ZIP
project:error
agentId, agentType, error, retryable (boolean) — nodo cambia a rojo con shake animation
project:paused
layer, layerName, lastFile — nodos atenúan animación, activa botón Continuar
07
Seguridad
Medidas de seguridad implementadas transversalmente en todos los módulos.
Autenticación
● JWT access token en cookie httpOnly Secure SameSite=Strict (15 min)
● Refresh token rotativo en cookie httpOnly (24h / 30d con rememberMe)
● bcryptjs cost 12 para hashing de contraseñas
● Mensaje genérico en login fallido (no revela si email existe)
Encriptación
● API keys: AES-256-GCM con IV único (12 bytes) por registro
● ENCRYPTION_KEY en .env (32 bytes hex, nunca en código ni BD)
● Auth tag 16 bytes almacenado separado del ciphertext
● Solo últimos 4 chars visibles al usuario (sk-ant-...XXXX)
Rate Limiting
● Login: 5 intentos / 15 min por email (Redis TTL)
● Registro: 3 por IP por hora
● Password reset: 3 por email por hora
● API key verify: 5 por usuario por hora
● Spec generation: 10/proyecto/hora, 50/usuario/día
Path Traversal Prevention
● Tools createFile/readFile validan ruta dentro de {PROJECTS_BASE}/{projectId}/
● No hay tool runCommand — agentes no ejecutan comandos
● Límites: 100 archivos/agente, 100KB/archivo, 5min timeout
● WebSocket auth via JWT en handshake + ownership check

12
Flujo de Ejecución de un Agente
Detalle del Tool Use loop que cada agente ejecuta dentro de su capa. El agente no genera texto — crea archivos iterativamente hasta completar su tarea.
1
Orchestrator asigna capa
Worker BullMQ toma job de la cola. Orchestrator selecciona la siguiente capa pendiente (1→6) y crea el agente correspondiente.
Datos de entrada
projectId, spec (content JSONB), archivos generados por capas anteriores (contexto acumulado)
2
Construye prompt con contexto
Lee skills/{agent}/system.md + task.md. Inyecta el spec del proyecto, data-model, api-design y lista de archivos ya generados.
Prompt structure
system: skill system.md
user: spec + context + task.md
tools: [createFile, readFile, listFiles, taskComplete]
3
Llamada a Claude API
POST /messages con tools definition. Claude analiza el contexto y decide qué tool usar primero.
Response type
stop_reason: "tool_use"
content: [{type: "tool_use", name: "createFile", input: {path, content}}]
4
Ejecuta tool en backend
Backend valida path (anti-traversal), ejecuta la operación en filesystem, registra en BD (generated_files) y emite WS event.
Tools disponibles
createFile(path, content) → escribe archivo
readFile(path) → lee archivo existente
listFiles(dir?) → lista directorio
taskComplete(summary) → finaliza agente
5
Retorna resultado a Claude
El resultado del tool se envía como tool_result. Claude decide si necesita otra operación o si ya terminó.
Loop condition
Repite pasos 3→4→5 hasta que Claude llame taskComplete o se alcance el timeout (5 min) o límite (100 archivos).
6
Agente completa / error
Si taskComplete → agent status=done, emite agent:status WS. Si error/timeout → status=error, project puede reintentar.
Post-completion
Orchestrator actualiza progress, avanza a siguiente capa. Al completar Layer 6 → project:done.
TOOL USE LOOP (pasos 3 → 4 → 5)
Claude API call → tool_use response → execute tool → tool_result → Claude API call → … → taskComplete

Escalabilidad y Evolución
Decisiones de escalabilidad para el MVP y roadmap de mejoras post-MVP.
MVP — Actual
Single Worker Process
Un proceso BullMQ worker con concurrency=3. Suficiente para ~50 usuarios concurrentes generando proyectos. Bottleneck principal: llamadas a Anthropic API (~30s por agente).
MVP — Actual
Sequential Layers
6 capas ejecutadas una tras otra. Tiempo total estimado por proyecto: 3-8 minutos. Sin paralelismo para evitar conflictos entre archivos generados.
Post-MVP
Horizontal Workers
Escalar agregando más procesos worker en Railway. BullMQ distribuye jobs automáticamente. Cada worker con su propia conexión a filesystem (Railway volumes compartidos o S3).
Post-MVP
Partial Parallelism
QA Agent (Layer 4) podría correr en paralelo con Docs Agent (Layer 5) ya que no modifican los mismos archivos. Deploy Agent (Layer 6) siempre último.
Post-MVP
S3 File Storage
Migrar archivos generados de filesystem local a S3/R2. Permite workers sin estado y descarga directa. BD mantiene path S3 en vez de path local.
Post-MVP
Streaming Token Billing
API key del usuario → billing directo. Tracking en tiempo real de tokens_input + tokens_output con pricing table (ANTHROPIC_PRICING). Dashboard de costos por proyecto.
